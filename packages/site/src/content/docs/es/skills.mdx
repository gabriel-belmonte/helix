# Skills

Una **skill** es una carpeta con un `SKILL.md` que contiene *instrucciones
especializadas* para un tipo concreto de tarea. Las skills **no son tools** —
son guía que el agente trae al contexto cuando una tarea encaja.

El diseño imita a OpenCode / Hermes: una lista compacta de skills disponibles
aparece en el system prompt, y el agente llama a `use_skill(name)` para cargar
las instrucciones completas cuando hace falta.

## Formato

```
skills/
  my-skill/
    SKILL.md
```

`SKILL.md` usa un frontmatter YAML mínimo (`name` + `description`) seguido de
las instrucciones:

```markdown
---
name: my-skill
description: Úsala cuando el usuario pregunte sobre X, Y o Z
---

# My Skill

Flujo de trabajo / conocimiento de dominio detallado que el agente debe seguir.
```

- `name` — cómo referencia el agente la skill (y lo que recibe `use_skill`).
- `description` — la pista de autoselección que el modelo lee para decidir
  *cuándo* cargar la skill. Escríbela para el modelo, no para humanos.

## Dónde viven las skills

Helix escanea estas carpetas automáticamente (global + proyecto, Helix-local y
compatibles con OpenCode/Claude):

- `~/.helix/skills` — tus skills personales
- `./skills` — skills del proyecto (commiteadas con el repo)
- `~/.claude/skills`, `./.claude/skills` — compatibles con OpenCode / Claude Code
- `~/.agents/skills`, `./.agents/skills` — compatibles con agent

Así, una skill que OpenCode instale en `.claude/skills/<name>` también la
reconoce Helix — sin mover nada.

Puedes añadir más vía `buildAgent({ skillDirs: [...] })`.

## Cómo funciona

1. Al arrancar, Helix descubre cada `SKILL.md` y renderiza un bloque
   `<available_skills>` en el system prompt.
2. Cuando una tarea encaja con una descripción, el agente llama `use_skill(name)`.
3. `use_skill` devuelve el body de la skill, que el agente entonces sigue.

La tool `use_skill` se añade automáticamente — no la registras tú.

## Uso programático

```ts
import { buildAgent } from "helix-core";
import { myLLM } from "./provider";

const agent = await buildAgent(myLLM, {
  config: {},
  skillDirs: ["./skills", "/path/to/more/skills"],
});
```

## Instalar skills desde skills.sh / OpenCode

Helix es compatible con el **Open Agent Skills Ecosystem**
([skills.sh](https://skills.sh)) — el mismo formato `SKILL.md` que usan
OpenCode, Claude Code y Cursor. Eso significa que puedes usar skills publicados
ahí sin cambios.

El formato es idéntico: una carpeta con un `SKILL.md` cuyo frontmatter YAML
tiene `name` + `description` (más `license`, `compatibility`, `metadata`
opcionales). Los campos de frontmatter desconocidos se ignoran, igual que en
OpenCode.

### Instalar una skill desde skills.sh

`skills.sh` es un índice sobre repos de GitHub. La instalación más simple es
clonar el repo fuente (el CLI `npx skills add` es interactivo):

```bash
# ejemplo: skill de buenas prácticas React de Vercel
git clone --depth 1 https://github.com/vercel-labs/agent-skills.git /tmp/agent-skills
mkdir -p ~/.helix/skills
cp -r /tmp/agent-skills/skills/react-best-practices ~/.helix/skills/
rm -rf /tmp/agent-skills
```

Helix la lista en `<available_skills>` y puede cargarla con `use_skill`.

> **Nota sobre nombres:** el agente llama a una skill por su `name` del
> frontmatter (p.ej. `vercel-react-best-practices`), no por el nombre de la
> carpeta. El system prompt muestra el nombre correcto, así el modelo lo usa
> tal cual.

### Compatibilidad de nombre de tool

Helix registra el loader como **`use_skill`** y también como **`skill`** (un
alias que coincide con OpenCode), así las skills escritas para llamar
`skill({ name })` funcionan sin cambios.
