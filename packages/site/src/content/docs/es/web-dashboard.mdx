# Dashboard (Web)

Helix incluye un **dashboard** web — un panel de control (al estilo Hermes)
para gestionar tu agente sin tocar la terminal. Es una UI fina sobre el mismo
estado en disco que usa la CLI, así que los cambios aquí se reflejan en todas
partes.

## Secciones

- **Chat** — habla con Helix en vivo desde el navegador. Usa el provider
  configurado (de `~/.helix/auth.json` o env). Sin key, corre en **modo demo**
  (un agente scripted) para ver el loop sin API key.
- **Config** — provider, model y base URLs (`~/.helix/config.json`). El
  campo **Model** es un dropdown de los modelos de OpenCode Zen, con los
  **gratuitos resaltados en verde** (Zen los marca con sufijo `-free`).
- **API Keys** — listar / añadir / quitar keys (`~/.helix/auth.json`, chmod 600).
  Enmascaradas con fingerprint sha256; las env vars siguen ganando.
- **Skills** — skills descubiertas en `~/.helix/skills`, `./skills`,
  `~/.claude/skills`, `~/.agents/skills` (compatibles con skills.sh / OpenCode).
- **MCP** — servidores de `~/.helix/helix.mcp.json`.
- **Files** — explorar la raíz del proyecto.

## Ejecutarlo

La forma más rápida — desde la CLI:

```bash
helix dashboard    # lanza el Dashboard web en http://localhost:8799
```

El binario incluye el servidor del Dashboard — no necesita Docker. Se ejecuta
en el mismo proceso y accede directamente a `~/.helix/` (skills, auth, MCP,
config funcionan todos).

O ejecútalo manualmente:

```bash
cd packages/web
bun install
bun run dev      # UI de Vite en :5173, API en :8799 (proxied)
```

O sirve el build de producción en un solo puerto:

```bash
bun run build    # genera dist/
bun run start    # sirve UI + API en :8799
```

Luego abre http://localhost:8799.

## Configura tu API key desde la web

**No** necesitas un provider configurado para abrir el Dashboard — arranca en
**modo demo** (un agente scripted) hasta que se setea una key. Para añadir tu
key:

1. Abre la sección **API Keys**.
2. Escribe el provider (ej. `zen`) y tu API key (el campo es `password`,
   enmascarado).
3. Clic en **Save**. La key se escribe en `~/.helix/auth.json` (chmod 600).

El indicador de **Status** cambia de `○ not set` a `● set` al instante, y la
sección **Chat** ahora habla con el modelo real. Las env vars
(`OPENCODE_ZEN_API_KEY`, etc.) siempre tienen prioridad sobre las keys
guardadas.

Este es el mismo flujo que usa la CLI — `helix auth login <provider>` escribe
el archivo idéntico — así que una key añadida en la web es visible en
`helix status` en la terminal, y viceversa.

## Arquitectura

`helix-web` es un proceso mínimo (Hono + React/Vite):

- **API** (`server/index.ts`) — REST sobre los archivos reales de Helix:
  config, auth, skills, MCP, files. Cada endpoint llama a `helix-core` /
  `helix-mcp` / los helpers de config de la CLI.
- **UI** (`src/`) — SPA de React con sidebar y un componente por sección.

Un solo puerto sirve tanto el dashboard como la API JSON.
