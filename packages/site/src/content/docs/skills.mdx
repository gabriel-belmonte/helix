# Skills

A **skill** is a folder with a `SKILL.md` file that holds *specialized
instructions* for a specific kind of task. Skills are **not tools** — they are
guidance the agent pulls into context when a task matches.

The design mirrors OpenCode / Hermes: a compact list of available skills is
shown in the system prompt, and the agent calls `use_skill(name)` to load the
full instructions when relevant.

## Format

```
skills/
  my-skill/
    SKILL.md
```

`SKILL.md` uses a tiny YAML frontmatter (`name` + `description`) followed by the
instructions:

```markdown
---
name: my-skill
description: Use when the user asks about X, Y, or Z
---

# My Skill

Detailed workflow / domain knowledge the agent should follow.
```

- `name` — how the agent references the skill (and what `use_skill` takes).
- `description` — the self-selection hint the model reads to decide *when* to
  load the skill. Write this for the model, not for humans.

## Where skills live

Helix scans these directories automatically (global + project, Helix-local and
OpenCode/Claude-compatible):

- `~/.helix/skills` — your personal skills
- `./skills` — project-local skills (committed with the repo)
- `~/.claude/skills`, `./.claude/skills` — OpenCode / Claude Code compatible
- `~/.agents/skills`, `./.agents/skills` — agent-compatible

So a skill installed by OpenCode into `.claude/skills/<name>` is picked up by
Helix too — no moving required.

You can add more via `buildAgent({ skillDirs: [...] })`.

## How it works

1. At startup, Helix discovers every `SKILL.md` and renders an
   `<available_skills>` block into the system prompt.
2. When a task matches a description, the agent calls `use_skill(name)`.
3. `use_skill` returns the skill's body, which the agent then follows.

The `use_skill` tool is added automatically — you don't register it yourself.

## Programmatic use

```ts
import { buildAgent } from "helix-core";
import { myLLM } from "./provider";

const agent = await buildAgent(myLLM, {
  config: {},
  skillDirs: ["./skills", "/path/to/more/skills"],
});
```

## Installing skills from skills.sh / OpenCode

Helix is compatible with the **Open Agent Skills Ecosystem** ([skills.sh](https://skills.sh))
— the same `SKILL.md` format used by OpenCode, Claude Code, and Cursor.
That means you can use skills published there with zero changes.

The format is identical: a folder with a `SKILL.md` whose YAML frontmatter has
`name` + `description` (plus optional `license`, `compatibility`, `metadata`).
Unknown frontmatter fields are ignored, exactly like OpenCode.

### Install a skill from skills.sh

`skills.sh` is an index over GitHub repos. The simplest install is to clone
the source repo (the `npx skills add` CLI is interactive):

```bash
# example: Vercel's React best-practices skill
git clone --depth 1 https://github.com/vercel-labs/agent-skills.git /tmp/agent-skills
mkdir -p ~/.helix/skills
cp -r /tmp/agent-skills/skills/react-best-practices ~/.helix/skills/
rm -rf /tmp/agent-skills
```

Helix now lists it in `<available_skills>` and can load it with `use_skill`.

> **Note on names:** the agent calls a skill by its frontmatter `name`
> (e.g. `vercel-react-best-practices`), not the folder name. The system
> prompt shows the correct name, so the model uses it as-is.

### Tool name compatibility

Helix registers the loader as **`use_skill`** and also as **`skill`** (an alias
matching OpenCode), so skills authored to call `skill({ name })` work unchanged.
