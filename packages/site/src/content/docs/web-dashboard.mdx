# Dashboard (Web)

Helix ships a web **dashboard** — a control panel (à la Hermes) for managing
your agent without touching the terminal. It is a thin UI over the same on-disk
state the CLI uses, so changes here are reflected everywhere.

## Sections

- **Chat** — talk to Helix live in the browser. Uses your configured provider
  (from `~/.helix/auth.json` or env). With no key set, it runs in **demo mode**
  (a scripted agent) so you can see the loop without any API key.
- **Config** — provider, model, and base URLs (`~/.helix/config.json`). The
  **Model** field is a searchable dropdown of OpenCode Zen models, with
  **free models highlighted in green** (Zen marks them with a `-free` suffix).
- **API Keys** — list / add / remove keys (`~/.helix/auth.json`, chmod 600).
  Masked with a sha256 fingerprint; env vars still win.
- **Skills** — skills discovered from `~/.helix/skills`, `./skills`,
  `~/.claude/skills`, `~/.agents/skills` (skills.sh / OpenCode compatible).
- **MCP** — servers from `~/.helix/helix.mcp.json`.
- **Files** — browse the project root.

## Run it

The fastest way — from the CLI:

```bash
helix dashboard    # launches the web Dashboard on http://localhost:8799
```

The binary bundles the Dashboard server — no Docker needed. It runs in the same
process and accesses `~/.helix/` directly (skills, auth, MCP, config all work).

Or run it from a source checkout:

```bash
cd packages/web
bun install
bun run dev      # Vite UI on :5173, API on :8799 (proxied)
```

Or serve the production build on a single port:

```bash
bun run build    # emits dist/
bun run start    # serves UI + API on :8799
```

Then open http://localhost:8799.

> **Port note:** `:8787` is used by Hermes' local Firecrawl on this machine, so
> Helix web defaults to `:8799`. Override with `PORT=... bun run start`.

## Configure your API key from the web

You do **not** need a provider configured to open the Dashboard — it starts in
**demo mode** (a scripted agent) until a key is set. To add your key:

1. Open the **API Keys** section.
2. Type the provider (e.g. `zen`) and your API key (the field is `password`
   masked).
3. Click **Save**. The key is written to `~/.helix/auth.json` (chmod 600).

The **Status** indicator turns from `○ not set` to `● set` immediately, and the
**Chat** section now talks to the real model. Env vars (`OPENCODE_ZEN_API_KEY`,
etc.) always take precedence over stored keys.

This is the same flow the CLI uses — `helix auth login <provider>` writes the
identical file — so a key added in the web is visible to `helix status` in the
terminal, and vice versa.

## Architecture

`helix-web` is one minimal process (Hono + React/Vite):

- **API** (`server/index.ts`) — REST over Helix's real files: config, auth,
  skills, MCP, files. Each endpoint calls `helix-core` / `helix-mcp` / the CLI
  config helpers.
- **UI** (`src/`) — React SPA with a sidebar and one component per section.

One port serves both the dashboard and the JSON API.
